RSA attacks & crypto math reference
Key insight: RSA challenges almost always exploit a weakness in key generation, not the math. Check these in order:
Small NFactor directly via FactorDB
Common factorGCD(N1, N2) gives shared prime p
Small ee=3, small msg → cube root attack
Wiener's attackSmall d → continued fractions
Fermat's methodp, q close together → fast factor
p,q givenφ(N)=(p-1)(q-1), d=e⁻¹ mod φ(N)
Hash extensionAppend data + valid hash, no key needed
ECB modeIdentical blocks → identical ciphertext
Decrypt formula: c^d mod N, where d = e⁻¹ mod φ(N) and φ(N) = (p-1)(q-1)
How to identify unknown ciphertext
1
Check for = or == at the end → likely Base64
2
All letters, no numbers or symbols → Caesar, Vigenère, or Atbash
3
Only 0-9 and a-f, 32/40/64/128 chars → MD5 or SHA hash
4
Dots and dashes (· -) → Morse code
5
Looks like gibberish English → try ROT13 first
6
Paste into CyberChef and use "Magic" mode, it auto-detects most encodings

Binary Exploitation & Pwn

Exploiting memory-safety bugs in compiled programs, buffer overflows, ROP chains, format strings, and shellcode. You're usually given a binary and a remote nc connection.

Triage First commands to run
checksecchecksec --file=binary
fileConfirm architecture and type
stringsLook for format strings, hints
ROPgadgetFind usable gadgets for chains
Mitigations What checksec output means
No NXStack is executable → shellcode works
No PIEAddresses fixed, easier ROP
No canaryStack smashing without detection
Partial RELROGOT is writable → GOT overwrite
Common attack patterns
1
Buffer overflow: Use cyclic(100) to find the offset, overwrite return address
2
ret2win: Overwrite return address to jump straight to a win function
3
ret2libc: Leak a libc address, identify version via libc-database, calculate system() + /bin/sh
4
Format string: %x/%n leaks or writes memory via printf-style bugs
5
ROP chain: Chain gadgets to set up registers and call functions with NX enabled

Reverse Engineering

Understanding what a compiled program does without source code, disassembly, decompilation, dynamic tracing, and unpacking. Often the path to a flag is buried in obfuscated logic.

Standard RE workflow
1
Run file binary and strings binary | grep -i flag first
2
Run Detect-It-Easy to check for packers (UPX etc.), unpack before analyzing
3
Open in Ghidra/IDA, read the decompiled main() first, rename variables as you understand them
4
Dynamically trace with ltrace/strace or GDB to watch comparisons like strcmp(input, secret)
5
For complex input validation, model constraints in angr or z3 and let the solver find the input
By platform Choosing your tool
Linux ELFGhidra, IDA Free, Radare2
.NET / C#dnSpy, full source, debuggable
Android APKjadx, apktool
Windows PEx64dbg, IDA, Binary Ninja
Quick wins Don't miss these
Strings firstMany crackmes leak flags in plain strings
Cross-referencesFind where the "wrong password" string is used
Unpack firstUPX-packed binaries show garbage until unpacked
Patch & runSometimes patching a jump is faster than understanding it

Wireless

Wi-Fi auditing, handshake capture, PMKID attacks, and RF/SDR challenges. Less common in CTFs but appears in dedicated wireless categories.

WPA2 handshake capture workflow
1
airmon-ng start wlan0, enable monitor mode
2
airodump-ng wlan0mon, find target network and BSSID
3
airodump-ng -c CH --bssid BSSID -w capture wlan0mon, capture
4
aireplay-ng -0 1 -a BSSID wlan0mon, deauth to force handshake
5
aircrack-ng -w rockyou.txt capture.cap, crack
No client needed: Use hcxdumptool to capture PMKID, then hashcat -m 22000, no deauth required.

Reference: Cheatsheets

Cheatsheets & Checklists

Quick-reference definitions for the basics across every category, plus a universal checklist to run through on any new challenge.

Universal challenge checklist
Run through this before going deep on any challenge. Most "impossible" challenges are solved by step 1-4.
1
Read the challenge title and description twice, the wording is usually a hint toward the category or technique
2
Run file on every provided file, never trust the extension
3
Run strings <file> | grep -iE "flag|ctf|key|pass" on every file
4
Check metadata with exiftool on any image/document/PDF
5
If it's a binary, run checksec and open in Ghidra alongside dynamic testing
6
If it's text/ciphertext, paste into this site's Multi-Decoder and CyberChef Magic
7
If it's a network capture, open in Wireshark and Follow every TCP/UDP stream
8
If stuck after 15-20 minutes, search "<category> <keyword from description> CTF writeup"
9
Note every tool/technique tried, this prevents repeating dead ends and builds your personal knowledge base

Common "rabbit hole" traps
These waste your time. Move on when you hit them.
Wrong toolNo output after 2 tries? Move on, don't force it
Over-encodingBase64 of Base64 is possible, keep layering if still gibberish
Noise pixelsNot every pixel variation is LSB steganography
Wrong keyIf output is gibberish after decoding, you likely have the wrong cipher/key
